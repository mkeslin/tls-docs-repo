# Customer implementation workbook

**Audience:** Your agency’s project team · Thin Line implementation team  
**Use:** Meetings and collaborative working sessions during implementation  

---

## How to use this workbook

This is a **shared working framework**, not a test and not an exhaustive intake form you must finish in one sitting.

- Complete sections **over time**, when Thin Line is ready to use the answers.  
- Mark a section **Not applicable** when it does not apply to your agency or purchase.  
- Prefer **defaults plus exceptions** (“we use the standard numbering unless…”) over filling every blank.  
- Separate **who we contact** from **who needs a login**.  
- Separate **decisions you make** from **facts you provide**.  
- Separate **project tasks** (“send logo by Friday”) from **software settings** Thin Line configures with you.  

Thin Line will tell you which sections are next. You should not feel pressure to answer technical questions that are not your job.

---

## Principles we work by

1. Ask for information only when we are ready to use it.  
2. Discover needs through **workflow conversations**, not only forms.  
3. Make **ownership and next steps** explicit after every session.  
4. Keep customer delay and Thin Line delay visible and distinct.  
5. Acceptance and go-live decisions come from people your agency authorizes.  

---

## Section guide

Each section below includes: why it matters, when we collect it, who provides it, when it is filled in, supporting files, and which implementation phase uses it.

| When filled | Meaning |
|-------------|---------|
| **Before the meeting** | Useful to gather ahead of time |
| **During a working session** | Best done together with Thin Line |
| **Later** | Captured after related work starts |

---

## 1. Project overview

**Why needed:** Shared picture of what you purchased, what “success” looks like, and what is in or out of scope.

| | |
|--|--|
| **When collected** | Early—kickoff and shortly after |
| **Who provides it** | Customer project lead · executive sponsor · Thin Line implementation lead |
| **When completed** | During a working session (kickoff); refine later if scope changes |
| **Supporting files** | Proposal or order summary (if you have it); nothing technical required |
| **Consumed by** | Kickoff and discovery · all later phases for scope checks |

**Typical prompts (not a quiz):**

- Products / modules in this project  
- What “live” means for your agency  
- Known out-of-scope items  
- Constraints (elections, fiscal year, facility moves, staffing)  

---

## 2. Implementation timeline

**Why needed:** Align calendars so training, cutover, and blackout dates do not surprise either side.

| | |
|--|--|
| **When collected** | Kickoff; update as dates firm up |
| **Who provides it** | Customer project lead · Thin Line implementation lead |
| **When completed** | During a working session; update later as needed |
| **Supporting files** | Agency calendar of blackout dates (optional) |
| **Consumed by** | Kickoff · training · go-live · hypercare |

**Typical prompts:**

- Target go-live window (or “not sure yet”)  
- Hard dates that cannot move  
- Preferred training days/times  
- Parallel-run or cutover preferences in plain language  

---

## 3. Customer and agency information

**Why needed:** Correct names, identity, URL preference, and how your agency should appear in the system and on documents.

| | |
|--|--|
| **When collected** | Kickoff for basics; report headers when configuration starts |
| **Who provides it** | Project lead · records / admin staff as needed |
| **When completed** | Basics before or during kickoff; document headers later |
| **Supporting files** | Letterhead or logo files; official agency name as you want it printed |
| **Consumed by** | Infrastructure (naming, preferred URL) · configuration (agency profile, reports, documents) |

**Typical prompts:**

- Legal / customer name  
- Agency display name and abbreviation  
- County / state · ORI (law enforcement)  
- Preferred site address (e.g. `youragency.thinline.app`) if you have one—otherwise Thin Line assigns one  
- How the agency should appear on printed reports (often the same as the identity above)  

We will not ask you to invent server names or cloud settings. Logo delivery is a **file task**, not a long form field—send files when Thin Line asks.

---

## 4. Contacts and responsibilities

**Why needed:** Clear who decides, who coordinates, and who Thin Line should call—separate from who needs a user account.

| | |
|--|--|
| **When collected** | Kickoff; update when people change |
| **Who provides it** | Executive sponsor · project lead |
| **When completed** | During kickoff; refine later |
| **Supporting files** | Org chart (optional) |
| **Consumed by** | Kickoff · every phase for ownership · go-live approval |

**Typical roles to name (as applicable):**

- Executive sponsor  
- Project lead (one accountable coordinator)  
- Final acceptance / go-live authority  
- IT or technical contact  
- **CJIS compliance officer** (name)—if your agency designates one  
- Records / RMS, dispatch, court, jail, code enforcement leads as needed  
- Training coordinator  

**Reminder:** Contacts ≠ application users. Officers, dispatchers, and jailers who need logins are planned in section 10 (usually a roster spreadsheet, not this contact list).

---

## 5. Current systems and environment

**Why needed:** Understand what you use today so we know what replaces, integrates, or stays beside Thin Line.

| | |
|--|--|
| **When collected** | Kickoff discovery; deepen only if migration or integrations apply |
| **Who provides it** | Project lead · IT · module leads |
| **When completed** | During a working session; details later for migration/integrations |
| **Supporting files** | Vendor names; export samples only when Thin Line asks |
| **Consumed by** | Kickoff · data migration · integrations |

**Typical prompts (plain language):**

- Systems you use today for records, CAD, court, jail, evidence, etc.  
- What must keep running after go-live  
- Whether historical data must move into Thin Line  
- Preferred Thin Line site address (if not already captured in section 3)—Thin Line can assign one  

Avoid deep technical questionnaires unless IT is in the room and Thin Line has asked for specifics.

---

## 6. Workflow discovery

**Why needed:** Configuration should match how your agency works—not only how software menus are labeled.

| | |
|--|--|
| **When collected** | After kickoff, before or during configuration |
| **Who provides it** | Module leads · supervisors · project lead |
| **When completed** | **During working sessions** (conversation-first) |
| **Supporting files** | Sample forms, SOPs, or screenshots of today’s process (optional) |
| **Consumed by** | Configuration · training · go-live planning |

**How we work this section:**

We walk through real days: “A citation is written… then what?”  
We capture **decisions** (how you want it to work) separately from **facts** (how many copies you print today).

Mark module areas **Not applicable** when you did not purchase them.

---

## 7. Data migration discovery

**Why needed:** Decide whether historical data moves, what “good enough” means, and who will validate it.

| | |
|--|--|
| **When collected** | Only if migration is in scope—when Thin Line is ready to plan the conversion |
| **Who provides it** | Project lead · records / module leads · IT for access/exports |
| **When completed** | Working sessions; file delivery later when requested |
| **Supporting files** | Legacy exports or access **when Thin Line requests them**—not before |
| **Consumed by** | Data migration · configuration (numbering alignment) · acceptance |

**Typical prompts:**

- Which history must be in Thin Line on day one  
- Who will review converted data  
- Known data-quality issues in the old system  
- **Data that must not be migrated** or that needs special handling / redaction (legal, juvenile, sealed, etc.)  

If you are not migrating history, mark this section **Not applicable**. Firm scope and pricing still require a formal migration assessment when migration is in play.

---

## 8. Configuration decisions

**Why needed:** Record the choices that set up your agency in Thin Line (identity, modules, numbering preferences, document headers, and similar).

| | |
|--|--|
| **When collected** | During configuration—not all at kickoff |
| **Who provides it** | Module leads · records admin · project lead (decisions) |
| **When completed** | During working sessions; some items later as modules are enabled |
| **Supporting files** | Logo, seal, letterhead images; preferred report headers |
| **Consumed by** | Configuration |

**Style:** Defaults plus exceptions. Thin Line proposes starting points; you confirm or adjust.

This section is for **decisions**, not a dump of every code table. Detailed settings—including numbering, NIBRS validation, court text, sharing, property locations, and notifications—live in the [Configuration discovery workbook](configuration-discovery-workbook.md).

**Do not** expect a multi-page call-type checklist. For CAD call types (and similar lists): use Thin Line defaults, import from legacy when converting, and list **exceptions** only.

---

## 9. Integrations and hardware

**Why needed:** Plan connections to other systems and devices (printers, scanners, etc.) so go-live is not blocked.

| | |
|--|--|
| **When collected** | When integrations or hardware are in scope—often after basic configuration is underway |
| **Who provides it** | IT · project lead · vendors as needed |
| **When completed** | Mix: inventory before a session; testing during sessions; credentials later under IT control |
| **Supporting files** | Device lists; vendor contact info; credentials shared securely when requested |
| **Consumed by** | Integrations and hardware · go-live |

Mark **Not applicable** if you have no integrations or special hardware in this project.

We will not ask non-IT staff for firewall rules or API secrets.

---

## 10. Users and access

**Why needed:** Plan who needs access and at what level—without confusing that list with project contacts.

| | |
|--|--|
| **When collected** | Before training and go-live; start listing roles earlier if helpful |
| **Who provides it** | Project lead · training coordinator · supervisors |
| **When completed** | Before training (rosters); refine later |
| **Supporting files** | **Personnel roster spreadsheet** (officers / dispatchers / jailers / clerks): name, email, badge/unit as applicable, supervisor, whether they approve reports |
| **Consumed by** | Configuration (accounts/roles) · training · go-live |

**Typical prompts:**

- Roles that need access (officer, clerk, dispatcher, jailer, supervisor, …)  
- Approximate counts for the go-live cohort  
- Who may approve reports or administer the site  

Contacts from section 4 are not automatically users. Do not paste large rosters into meeting notes—attach the spreadsheet.

## 11. Training planning

**Why needed:** Right people, right sessions, before exclusive production use.

| | |
|--|--|
| **When collected** | After major workflows and configuration direction are clear |
| **Who provides it** | Training coordinator · project lead · Thin Line training owner |
| **When completed** | Planning session; attendance recorded during/after training |
| **Supporting files** | Shift calendars; trainee lists |
| **Consumed by** | Training · go-live readiness |

**Typical prompts:**

- Audiences by job role  
- Preferred format (on-site, remote, multiple shifts)  
- Who must be trained before go-live vs afterward  

---

## 12. Go-live planning

**Why needed:** A controlled cutover—not a surprise “flip the switch” day.

| | |
|--|--|
| **When collected** | As go-live approaches; outline earlier if the agency needs it |
| **Who provides it** | Project lead · acceptance authority · supervisors · Thin Line implementation lead |
| **When completed** | Working sessions near go-live |
| **Supporting files** | Duty schedules; public or internal communication drafts (optional) |
| **Consumed by** | Go live · hypercare |

**Typical prompts:**

- Cutover date/window  
- What stops in the old system and when  
- Who is on duty for Thin Line and for your agency  
- How you will call for help in the first days  

---

## 13. Open questions and action items

**Why needed:** Explicit next steps and owners so work does not disappear into email.

| | |
|--|--|
| **When collected** | Every working session |
| **Who provides it** | Everyone present; Thin Line records the board |
| **When completed** | During and after each meeting |
| **Supporting files** | None required |
| **Consumed by** | Entire project (workspace / action list) |

For each item, capture:

- What  
- Owner (one person)  
- Due date if it matters  
- Waiting on customer vs waiting on Thin Line

## Mapping to the Thin Line Implementation Workspace (future)

Today this workbook guides conversations and notes. Over time, the same sections will map into a **customer-facing Thin Line Implementation Workspace** where your team and Thin Line can see shared status, owners, and files in one place.

| Workbook section | Future workspace area (conceptually) |
|------------------|--------------------------------------|
| Project overview · timeline | Project identity and schedule |
| Contacts and responsibilities | Contacts and roles |
| Current systems · workflow · migration · configuration · integrations · users · training · go-live | Phase-aligned working areas and decisions |
| Open questions and action items | Tasks, owners, and waiting-on status |
| Supporting files | Linked customer documents |

Internal Thin Line procedures stay internal. The workspace will share **progress and collaboration**, not engineering runbooks.

Until that workspace is available, Thin Line will keep an implementation record and share the parts you need—timelines, actions, and decisions—using the tools agreed for your project.
