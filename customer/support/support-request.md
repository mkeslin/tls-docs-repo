# What to include in a support request

Clear requests get faster answers. Include as much of the following as you can.

## Required context

| Item | Why it helps |
|------|----------------|
| **Agency name** | Multi-tenant systems need the correct tenant |
| **Your username** (not password) | Reproduce permissions and audit |
| **What you were trying to do** | Expected outcome in one or two sentences |
| **What happened instead** | Error text, unexpected blank screen, wrong data |
| **When it happened** | Date/time and time zone (Central if unsure) |
| **Where in the product** | Module (Court, Jail, Incidents, …), screen name, and steps |

## Strongly helpful

- Screenshot or short screen recording (no unnecessary PII)
- Record identifiers you can share under policy (court violation number, booking id, incident number)
- Browser name and version
- Whether the issue happens for one user or many
- Whether it started after a known change (new user, code change, go-live cutover)

## Do not send

- Passwords or temporary passwords in email or chat
- More personal data than needed to diagnose the issue
- Speculative “reinstall everything” unless Thin Line asks
